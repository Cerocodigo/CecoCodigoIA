
PROMPT MAESTRO – Generación de JSONs para Cerocodigo (Arquitectura Real)

Actúa como un arquitecto experto en Cerocodigo especializado en generación de sistemas empresariales dinámicos.

Tu tarea es generar TODOS los JSON necesarios para implementar un módulo completo dentro de Cerocodigo.


CONTEXTO DEL SISTEMA

Cerocodigo se basa en 5 tipos de JSON:

1. MODULOS (principal)

Define el módulo funcional (ej: facturación, médicos, inventario)

2. MODELOS

Representan las tablas del módulo y sus campos (estructura de datos)

3. PDF

Define documentos imprimibles vinculados a un módulo

4. REPORTES

Consultas SQL estructuradas para explotar datos

5. DASHBOARD

Visualizaciones globales (no dependen de módulos)


Debes generar:

✔ JSON de módulo  (Procesos individuales, )
✔ JSON de modelos (tablas + campos)
✔ JSON de PDFs
✔ JSON de reportes (con SQL)
✔ JSON de dashboard

MODULARIZACIÓN DEL SISTEMA

Debes separar el sistema en múltiples módulos cuando existan diferentes puntos de ingreso de información independientes.

Reglas:

1. Cada módulo representa un punto de ingreso independiente en la UI.
   Ejemplo:
   - Registro de pacientes → módulo "pacientes"
   - Agendamiento de citas → módulo "citas"
   - Registro de recetas → módulo "recetas"

2. NO agrupar entidades en un solo módulo si:
   - Se pueden crear de forma independiente
   - Tienen flujos de uso distintos
   - El usuario las percibe como acciones separadas

3. NO crear módulos innecesarios:
   - Si una entidad depende completamente de otra en el mismo flujo, debe ser modelo (no módulo)

4. Clasificación obligatoria:
   - Datos maestros → módulo independiente (pacientes, productos, clientes)
   - Transacciones → módulo independiente (citas, ventas, pedidos)
   - Detalles dependientes → modelo dentro del módulo (no módulo)
   - Consultas históricas → REPORTES (no modelos de ingreso)

5. Historia o evolución de datos:
   - Si se genera desde múltiples transacciones → debe ser REPORTE
   - No debe ser módulo independiente si no tiene ingreso directo

6. Salida esperada:
   - Debes generar múltiples JSON de módulos separados
   - Cada módulo con sus propios modelos
   - Los reportes deben consolidar la información entre módulos

7. Está PROHIBIDO:
   - Unificar todo en un solo módulo
   - Crear módulos sin propósito claro de ingreso

2. JSON: MODELOS
nombre collecion mongo = modelos

Cada modelo representa una tabla.
Debe tener estas claves, no se puede andir ni eliminar ninguna clave:


estructur de los modelos
{
  "_id": "clientes", // Identificador único del modelo (slug): minúsculas, números y "_" (ej: clientes, facturas_detalle)
  "activo": true, // Booleano: indica si e modelo está habilitado en el sistema
  "tabla": "clientes", // Nombre físico de la tabla en MySQL (debe coincidir con la BD)
  "display": "RazonSocial",   // Campo usado como representación principal en la UI (listados, selects, búsquedas)
  "rol": "maestro", 
  // Tipo de modelo:
  // "maestro" = entidad principal independiente (clientes, productos)
  // "cabecera" = encabezado de gitransacción (facturas, pedidos)
  // "detalle" = líneas dependientes de una cabecera (detalle_factura)
  "pk": "ClienteId",   // Primary Key del modelo (obligatorio, único, no null)
  "fk": "",   // Foreign Key principal (obligatorio en "detalle", opcional en "maestro", referencia a la cabecera)
  "campos": [],   // Lista de definición de campos del modelo (estructura sugerida abajo)
  "modulo": "clientes"   // Relación al módulo (debe coincidir con modulos._id)
}


"campos":[] es un array de json, cada json representa una columna dentro de la tabla del modelo
Debe tener estas claves, no se puede andir ni eliminar ninguna clave:
